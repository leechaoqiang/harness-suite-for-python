# 隐性业务约定

> 本文档记录项目中**没有显式文档化但前后端/各模块共同依赖**的约定。
> 修改前务必确认依赖方，必要时在 design.md 中明确影响范围。

---

## 示例约定 1：状态字段语义

- **背景**：前后端口头约定，`status = None` 表示「未初始化」，`status = 0` 表示「已关闭」
- **代码现状**：后端代码中没有统一显式注释，前端页面逻辑依赖这个区别
- **风险**：如果把 `None` 和 `0` 合并处理，可能导致页面状态判断错误
- **处理要求**：
  - 修改该字段相关逻辑前，先确认前端依赖
  - 不要擅自把 None 归一化成默认值
  - 如确需调整，必须在 OpenSpec design 中明确说明影响面

---

## 示例约定 2：API 返回结构

- **背景**：前后端口头约定，后端会将前端传入的 item_id 转化成完整对象返回
- **代码现状**：前端依赖返回中的嵌套对象进行展示
- **风险**：如果缺失嵌套对象前端会报错
- **处理要求**：
  - 修改该接口相关逻辑时，需要保留当前返回结构
  - 如确需调整，必须在 OpenSpec design 中明确说明影响面

---

## 如何添加新约定

在设计评审中，如果发现新的隐性依赖，应：

1. 在本文档中新增条目
2. 在对应的 service/model 代码中添加注释
3. 通知相关方（前端/其他团队）
